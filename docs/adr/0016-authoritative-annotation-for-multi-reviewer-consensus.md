# ADR-0016: Authoritative annotation for multi-reviewer consensus

## Context

[ADR-0015](0015-human-annotations-app-with-queue-item-annotation-aggregate-model.md) records that an `AnnotationQueue` carries a `num_reviews_required` between 1 and 10, and that an `AnnotationItem` can collect multiple submitted `Annotation` rows — one per reviewer. The downstream consumers (the score writer at [ADR-0013](0013-dual-write-scores-from-evaluations-and-annotations.md), the concordance view at [ADR-0014](0014-minimal-read-side-concordance-view.md), CSV/JSONL exports, the per-queue aggregate at [ADR-0017](0017-eager-aggregation-of-submitted-annotations.md)) need a single, unambiguous "this is the answer for this item" signal: which reviewer's judgment counts when more than one disagrees? The same question shapes the item-status state machine — an item with two of three required reviews is not the same state as an item where the queue admin has picked a winner, even if both have non-zero review counts. The design had to support single-reviewer queues without ceremony (the common case) and multi-reviewer queues with explicit, auditable resolution (the harder case), in one model.

## Decision

We will represent the consensus answer as a boolean `is_authoritative` flag on `Annotation`, enforced and resolved as follows:

- **At-most-one authoritative annotation per item, at the database layer.** `Annotation` carries `is_authoritative` plus an `authoritative_set_by` nullable FK to `AUTH_USER_MODEL` and an `authoritative_set_at` timestamp. The `one_authoritative_annotation_per_item` partial `UniqueConstraint` covers `(item)` where `is_authoritative=True`, so a malformed multi-authoritative state is impossible to write.
- **Single-reviewer queues auto-mark the first submission as authoritative.** When `num_reviews_required == 1` and no other authoritative annotation already exists on the item, the submitting reviewer's annotation gets `is_authoritative=True` and `authoritative_set_at=now` set inside the same transaction. `authoritative_set_by` is left null to distinguish auto-marking from a deliberate admin override.
- **Multi-reviewer queues require an explicit, permissioned set.** The "set authoritative" endpoint requires `human_annotations.change_annotationqueue` (queue-admin permission). It clears the flag on any sibling annotation before setting it on the chosen one — application-layer enforcement of the same single-winner invariant the constraint enforces at the DB layer, so the constraint is never the user-visible failure path.
- **Concurrent submissions are serialised by a row-level lock.** On every new submitted `Annotation`, the save path takes `select_for_update()` on the parent `AnnotationItem` before deciding whether to auto-mark and before incrementing `review_count`. Two concurrent submissions on a single-review queue cannot both pass the "no authoritative yet" check and both succeed — the second waits, observes the first's authoritative annotation, and skips the auto-mark.
- **Item status is derived from review count + authoritative flag.** `AnnotationItem.status` is recomputed on every annotation save into an `AnnotationItemStatus` enum: `PENDING` (`review_count == 0`), `IN_PROGRESS` (some reviews but below `num_reviews_required`), `AWAITING_RESOLUTION` (multi-reviewer queue at or above quota with no authoritative pick yet), `COMPLETED` (single-reviewer at quota, or any item with an authoritative annotation), or `FLAGGED`.
- **`FLAGGED` is sticky and only cleared by explicit unflag.** Once a reviewer flags an item, the recompute returns early without touching `status`. The unflag endpoint resets `status` to `PENDING` and re-runs the same `review_count`-based recompute to land it in the correct downstream state. This keeps flagged items visible in the queue admin's worklist regardless of how many reviews accumulate afterwards.

## Consequences

- **Positive:** The "what is the answer" question has one database-enforced signal across writers, readers, and exports. The score writer at [ADR-0013](0013-dual-write-scores-from-evaluations-and-annotations.md) deliberately writes Scores for every submitted annotation (not just authoritative ones) so inter-rater data is preserved; the concordance view at [ADR-0014](0014-minimal-read-side-concordance-view.md) and the aggregate at [ADR-0017](0017-eager-aggregation-of-submitted-annotations.md) filter to authoritative-or-fallback at read time.
- **Positive:** Auto-marking single-reviewer queues means the common case is zero-ceremony — most queues will never enter `AWAITING_RESOLUTION`, the admin never has to "pick a winner", and reviewer UX matches the single-rubric mental model.
- **Positive:** The row-level lock on `AnnotationItem` makes the constraint untrigger-able under normal load: by the time the second concurrent submission acquires the lock, the first has already committed, so the second observes the existing authoritative and skips. The DB constraint is the safety net, not the contention point.
- **Positive:** A sticky `FLAGGED` state means a reviewer can flag a session as broken (off-topic content, corrupted data) and the item stays surfaced for admin attention regardless of how the rest of the queue progresses; unflagging is explicit, audit-friendly, and returns the item to the recomputed natural state.
- **Negative:** `is_authoritative` is a derived signal that mixes "auto-set by the queue's single-review policy" with "explicitly picked by a queue admin", and only the `authoritative_set_by IS NULL` heuristic distinguishes them. Future "show me items whose authoritative answer was overridden" reporting needs to read the audit log of `SetAuthoritative` calls, not infer from this field.
- **Negative:** Multi-reviewer queues that the admin never resolves accumulate `AWAITING_RESOLUTION` items indefinitely — there is no auto-resolution by majority vote or by latest submission. This is deliberate (silent majority voting would surprise admins who chose multi-reviewer for a reason) but it requires admin discipline.
- **Negative:** Editing a submitted annotation keeps the same authoritative flag — a queue admin who picked a winner cannot have their pick silently overturned by the chosen reviewer revising their answer. This is correct for audit but means a reviewer revision after authoritative-set is invisible to the aggregate unless the admin re-picks.

## Alternatives considered

- **Implicit consensus by majority vote on multi-reviewer queues:** rejected — for categorical rubrics with no tie-breaker, majority is silent and easy to mis-tally; admins choosing multi-reviewer typically want adjudication, not voting. We can layer a "suggest majority" affordance on top of the explicit set later without changing the schema.
- **One row per consensus stored separately from per-reviewer rows:** rejected — would double the write path, force a separate constraint surface, and split the score-writer dispatch ([ADR-0013](0013-dual-write-scores-from-evaluations-and-annotations.md)) across two models. The flag-on-the-winning-row pattern keeps a single `Annotation` table as the source of truth and lets the DB enforce uniqueness at the same row.
- **A `consensus_annotation` FK on `AnnotationItem`:** rejected — symmetric to the flag in effect, but harder to enforce "exactly one of my children is the chosen one" than the partial unique constraint we already have. The FK direction also makes the "no consensus yet" state nullable rather than absent, adding a third tri-state for readers.
- **Resolve in a Celery task instead of inline:** rejected — single-reviewer auto-mark needs to be observable to the same request that submits the annotation (the score writer reads `is_authoritative` immediately, the item status recompute depends on it). A deferred resolution would force every reader to handle the in-between state.
- **Use `UniqueConstraint(condition=Q(is_authoritative=True))` without the application-layer pre-clear in `SetAuthoritative`:** rejected — the DB constraint would surface as an `IntegrityError` to the queue admin if they click "make authoritative" on a second annotation. Pre-clearing in the same transaction makes the admin action idempotent and presents the DB constraint as a backstop rather than a UX failure mode.
