# ADR-0015: Dedicated human_annotations app with queue/item/annotation/aggregate model

<span class="adr-status adr-status-accepted">ACCEPTED</span>

<p class="adr-meta">Author: Open Chat Studio Â· Created: 2026-05-29</p>

## Context

Structured human review of chat sessions is a recurring need across teams: labelling completed conversations against a shared rubric, gathering reviewer judgments at scale, exporting them for analysis, and comparing them to automated judgments. Before this work, `Tag` and `CustomTaggedItem` covered free-form session annotation but offered no schema, no reviewer assignment, no review-count tracking, and no aggregation surface. The dogfood pilot for multi-reviewer labelling needed a first-class data model that supports per-queue rubrics, controlled reviewer access, idempotent item membership, and progress tracking â€” and that does so as a peer to `apps.evaluations` (the automated judgment subsystem) rather than as an extension of the tags app, since the lifecycles, permissions, and consumers are different. The constraint shaping this ADR was therefore: model the queue/item/annotation hierarchy explicitly, reuse the existing schema vocabulary from evaluations rather than inventing a parallel one, and lock the rubric the moment it begins to carry real reviewer work so the data stays comparable.

## Decision

We will house human review in a dedicated `apps.human_annotations` Django app organised around four `BaseTeamModel` records:

- **`AnnotationQueue`** is the rubric + reviewer container. It holds a `schema` as a `SanitizedJSONField` mapping field names to `FieldDefinition` entries (the same Pydantic-discriminated union used by `apps.evaluations` for evaluator output schemas â€” int / float / choice / string); a `num_reviews_required` `PositiveSmallIntegerField` (1-10, enforced by a `num_reviews_required_range` `CheckConstraint`); a `status` from a `QueueStatus` enum (`ACTIVE` / `PAUSED` / `COMPLETED` / `ARCHIVED`); and an `assignees` `ManyToManyField` to `AUTH_USER_MODEL`. `(team, name)` is unique.
- **`AnnotationItem`** is the unit of work. `item_type` is an `AnnotationItemType` enum (`SESSION` / `MESSAGE`) backed by mutually-exclusive nullable FKs to `experiments.ExperimentSession` and `chat.ChatMessage`. Partial unique constraints (`unique_session_per_queue`, `unique_message_per_queue`) enforce one item per (queue, target). A denormalised `review_count` and an append-only `flags` `SanitizedJSONField` avoid recomputing per-render.
- **`Annotation`** is one reviewer's submission. `(item, reviewer)` is unique â€” one review per reviewer per item. `data` is a `SanitizedJSONField` whose keys must match the queue's `schema` (form-validated at submission time). `status` (`AnnotationStatus`: `DRAFT` / `SUBMITTED`) gates whether the row contributes to counts and aggregates.
- **`AnnotationQueueAggregate`** is a `OneToOneField` to `AnnotationQueue` that caches per-field aggregates (see [ADR-0017](0017-eager-aggregation-of-submitted-annotations.md)).

We also will:

- **Reuse `FieldDefinition` from `apps.evaluations` rather than fork it.** Queue rubrics and evaluator output schemas describe the same shape â€” typed scalar fields with required/optional + choices â€” so the same `TypeAdapter(FieldDefinition)` validates both. A single shared vocabulary makes [ADR-0012](0012-score-value-layer-in-apps-assessments.md)'s `Score` writes type-dispatch uniformly across automated and human paths.
- **Lock rubric structure once any item has received a review.** `AnnotationQueue.schema` and `num_reviews_required` become immutable as soon as `items.filter(review_count__gt=0).exists()`; only the per-field `required` flag may change. This protects the comparability of data already collected â€” adding, removing, or retyping a field mid-queue would silently invalidate prior submissions.
- **Make queue visibility role-based via a custom manager.** `AnnotationQueue.objects.visible_to(user, team)` returns all team queues for users with the `human_annotations.add_annotationqueue` permission (queue admins) and only the subset where `user in assignees` for everyone else. Annotation itself is gated by an in-view assignee check (`queue.assignees` empty = open to any team member; non-empty = restricted to the listed users), so a queue can be either invite-only or team-wide without a separate flag.
- **Gate user-facing entry points behind a Waffle flag.** `flag_human_annotations` hides the nav, the experiment-session "Add to queue" affordance, and the cross-cutting "add session from session detail" view. The queue management views themselves are not flag-gated; the flag controls discovery, not data access.

## Consequences

- **Positive:** A first-class data model means downstream consumers (the score writer at [ADR-0013](0013-dual-write-scores-from-evaluations-and-annotations.md), the concordance view at [ADR-0014](0014-minimal-read-side-concordance-view.md), CSV/JSONL exports) can rely on stable FK targets â€” `Annotation` is referenced by `assessments.Score.review` and survives the eventual `Annotation â†’ Review` rename without a schema-level shift.
- **Positive:** Sharing `FieldDefinition` means the same Pydantic validation, the same form widget mapping, and the same aggregation type-dispatch ([ADR-0017](0017-eager-aggregation-of-submitted-annotations.md)) cover both subsystems. New field types ship in one place.
- **Positive:** Schema-lock + per-field `required` mutability hits the right point on the rigidity/flexibility curve: queue admins can still tighten or relax which fields are mandatory after launch, but they cannot retroactively redefine what was being measured.
- **Positive:** Role-based `visible_to` means a single queue list page renders correctly for both admins and reviewers without separate URLs; assignees see exactly the work they're responsible for.
- **Negative:** Adding a fifth `BaseTeamModel` here means each team gains one more queryset to scope; the multi-tenancy invariant is preserved by inheritance from `BaseTeamModel` but every new view must remember to filter through `team=request.team` or `visible_to(...)`.
- **Negative:** Locking the schema on the first review (not the first submission, not the queue's `ACTIVE` flag) means a queue can sit in an editable state through the entire reviewer-assignment phase, only to lock at the moment work begins. The trigger is correct but easy to miss when editing â€” the form disables fields on render to make this visible.
- **Negative:** The `flags` `JSONField` on `AnnotationItem` is append-only and unsanitised beyond the `SanitizedJSONField` HTML strip; queries against flag contents require JSON operators. This is acceptable for an audit trail but not for filtered listings â€” `AnnotationItemStatus.FLAGGED` is the indexed signal for that.

## Alternatives considered

- **Extend `Tag` / `CustomTaggedItem` with a "queue" concept:** rejected â€” tags are flat strings without typed schemas, without per-reviewer uniqueness, and without an obvious place to hang a `review_count` or an aggregate record. Shoehorning a queue lifecycle on top would have produced a worse `Tag` and a worse queue.
- **Fold human review into `apps.evaluations` as a "manual evaluator":** rejected â€” evaluations are Celery-driven, idempotent re-runs against the same dataset; annotations are interactive, per-user, and can be edited. Sharing the app would have meant constantly disambiguating reviewer-vs-evaluator semantics in shared code.
- **Make `AnnotationItem` a `GenericForeignKey` over session-or-message:** rejected â€” only two target types are needed, the constraints (`unique_session_per_queue`, `unique_message_per_queue`) are clearer as partial unique indexes on dedicated FKs, and the prefetch ergonomics for the items table benefit from typed `select_related("session__experiment", "message")`.
- **Single global `Annotation` model shared across both automated and human paths:** rejected â€” automated results come from `EvaluationResult.output` with a different lifecycle (worker-driven, re-runnable per evaluator config) and would force a discriminator on every column. [ADR-0012](0012-score-value-layer-in-apps-assessments.md) already provides the shared value layer where the two paths converge.
- **Allow free-form schema edits at any time:** rejected â€” once a single reviewer has answered "did this session resolve the user's question? yes/no", silently letting the queue admin change "yes/no" to a 1-5 scale invalidates the existing submission with no migration path. Locking on first review is the conservative default; the per-field `required` escape hatch covers the only case (toggling a field's optionality) that doesn't change comparability.
